## Отчёт о качестве системы обнаружения

Документ содержит расчёт метрик Precision, Recall, F1 на основе сопоставления RAG-детекций с реальными событиями в логах.

---

## **1. Методология подсчета**

**Правила подсчёта:**

### 1.1. Что считается одной единицей измерения (one event)?

В анализе **каждое уникальное сопоставление** "RAG-техника → реальное событие в логе" считается как **1**.

**Пример:** Если в одном временном окне RAG содержит `T1112, T1543, T1190`, а в логах есть:
- `Registry modification` (→ T1112) ✅
- `Service state changed` (→ T1543) ✅
- Нет события под T1190 ❌

То:
- TP = 2
- FN = 1 (T1190 не подтвержден)

---

### 1.2. Как обрабатываются дублирующиеся инциденты?

В файле `pipeline_metrics_lastday2.log` для **одного временного отрезка** часто есть **2 инцидента с разными RAG-наборами**.

**Пример:** Отрезок `16:42:28-16:43:26`:
- Инцидент 1: `T1558, T1003, T1558, T1112`
- Инцидент 2: `T1558, T1190, T1059, T1112`

**Правило подсчета:** **Уникальные техники** из обоих инцидентов объединяются для этого временного окна, так как реальные события одни и те же.

**Объединенный RAG для 16:42:28-16:43:26:** `T1558, T1003, T1112, T1190, T1059`

---

### 1.3. Как определяется "реальное событие"?

Источник истины — `generated_logs_lastday2.log`. Каждая строка анализируется и сопоставляется с MITRE ATT&CK.

**Пример сопоставления:**

| Строка лога | Техника MITRE | Обоснование |
|-------------|---------------|-------------|
| `cmd.exe execution: asreproast` | T1558.004 | AS-REP Roasting |
| `File content viewed: /etc/passwd` | T1589.001 | Сбор учетных записей |
| `vssadmin.exe delete shadows` | T1490 | Удаление теневых копий |
| `Set-ExecutionPolicy Bypass` | T1070.004 | Отключение политик |
