## 3. Умный чат с роутингом

### Концепция

Пользователь отправляет запрос в чат → LLM определяет тип запроса → Выполняется соответствующее действие:

```
┌─────────────────────────────────────────────────────────┐
│                    Пользователь                         │
│  "Посмотри логи" / "Что за ошибка?" / "Покажи отчёт"   │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│              LLM Router (классификатор)                 │
│  "Это запрос на анализ логов ИЛИ вопрос по знаниям?"    │
└──────────────┬──────────────────────┬───────────────────┘
               │                      │
        ┌──────▼──────┐        ┌──────▼──────┐
        │   ЛОГИ      │        │  ВОПРОС     │
        │ (анализ)    │        │  (RAG)      │
        └──────┬──────┘        └──────┬──────┘
               │                      │
               ▼                      ▼
    ┌──────────────────┐   ┌──────────────────────┐
    │  Пайплайн:       │   │  RAG поиск:          │
    │  Vector → Kafka  │   │  - MITRE ATT&CK      │
    │  → GigaChat      │   │  - История инцидентов│
    │  → DB Report     │   │  - База знаний       │
    └──────────────────┘   └──────────────────────┘
```

### Варианты роутинга

| Вариант | Описание | Сложность | Время |
|---------|----------|-----------|-------|
| **LLM-классификатор** | Автоматическое определение типа запроса | 🟡 Средняя | 4-8 часов |
| **Явные команды** | `/analyze`, `/question` | 🟢 Низкая | 1-2 часа |
| **Гибридный** | LLM + уточняющие вопросы | 🔴 Высокая | 8-16 часов |

**Выбрано:** Гибридный подход

### Задачи

- [ ] **Создать цепочку классификации** (`ai_agent_v2/chains/router.py`)
  - Промпт для определения намерения пользователя
  - Категории: `LOG_ANALYSIS`, `MITRE_QUESTION`, `HISTORY_QUERY`, `CHAT`
  - Оценка уверенности классификации

- [ ] **Реализовать логику уточняющих вопросов**
  - Если уверенность < порога → задать уточняющий вопрос
  - Пример: "Вы хотите проанализировать логи или найти информацию в базе знаний?"

  - UI для отображения уточняющих вопросов

## 5. Автоматическая настройка

### Концепция

Скрипт "одного касания" для развёртывания системы:

```bash
# Пользователь запускает:
./setup.sh          # Linux/Mac
# или
.\setup.ps1         # Windows
# или
python setup.py     # Универсальный

# Скрипт выполняет:
# 1. 🔍 Поиск папок с логами рядом с проектом
# 2. ❓ Если не найдено — запрос пути вручную
# 3. 📝 Создание .env из .env.example
# 4. ⚙️ Проверка наличия Docker
# 5. 🚀 Запуск docker compose up -d
# 6. ⏳ Ожидание готовности сервисов
# 7. 🔧 Запуск CLI для регистрации пользователя
# 8. ✅ Показ инструкции
```

### Структура файлов

```
log_ai_agent/
├── setup.sh              # Bash-скрипт для Linux/Mac
├── setup.ps1             # PowerShell для Windows
├── setup.py              # Python-скрипт (универсальный)
├── docker-compose.yml    # С переменной ${EXTERNAL_LOGS_PATH}
```

## 📝 Примечания

### Конфигурация Vector

**Файл:** `log_ai_agent/vector/vector.toml`

- Батчи: 250 записей или 5 минут таймаут
- Overlap: 20 записей между батчами
- Эвристика определения лог-линий: timestamp, level, brackets

### Конфигурация Kafka

**Файл:** `log_ai_agent/docker-compose.yml`

- Режим: KRaft (без ZooKeeper)
- Топик: `processed-logs-batches` (авто-создание)
- Consumer group: `cyberlog-backend-group`

### Потенциальные проблемы

| Проблема | Решение |
|----------|---------|
| Vector читает все файлы | Настроить `include` паттерны |
| Повторное чтение после рестарта | `fingerprint.strategy = "device_and_inode"` |
| Нет логов в начале | `read_from = "beginning"` — прочитает всё |
| Права доступа к volume | `user: "0:0"` в Kafka и Vector |
