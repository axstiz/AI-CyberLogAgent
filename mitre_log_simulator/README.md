# MITRE Log Simulator

Этот каталог содержит отдельный Docker-симулятор для генерации логов в двух слоях:

1. Фоновый шум на базе Flog в Apache combined-формате.
2. Слой синтетических атак на основе YAML-описаний техник из `atomic-red-team`.

Симулятор **не выполняет реальные атаки**. Он читает YAML-файлы техник MITRE ATT&CK, анализирует команды и генерирует правдоподобные Apache syslog-строки, которые имитируют активность атакующего.

Симулятор пишет данные одновременно в stdout контейнера и в файловый лог внутри внешнего volume. Файлы дублируются в привязанную папку на хосте (`shared_external_logs`):

- `generated_logs.log` — все сгенерированные логи (шум + атаки)
- `attack_timeline.log` — таймлайн атак в формате `timestamp_start|timestamp_end|technique`
- `attack_markers.csv` — временные метки атак для сопоставления с анализом
- `unknown_techniques.txt` — техники, которые генератор не смог покрыть

### unknown_techniques.txt

Если ни одна команда из YAML-файла техники не подошла ни под один известный шаблон — техника **не попадает** в основной лог, таймлайн и маркеры, а записывается в `unknown_techniques.txt` (каждая с новой строки: `T1564.001`). Это позволяет оценить покрытие MITRE ATT&CK и показать заказчику, какие техники не были симулированы. Файл **не очищается** при перезапусках — только дополняется.

## Требования

- Docker Desktop или Docker Engine с поддержкой `docker compose`
- PowerShell 7 на хосте для запуска `run.ps1`
- Доступ в интернет на этапе сборки образа, чтобы скачать `atomic-red-team`

## Запуск

Базовый старт без атак:

```powershell
.\run.ps1 start -NoAttacks
```

Одна конкретная техника:

```powershell
.\run.ps1 start -Technique T1059 -IntervalSeconds 10
```

Рандомные атаки:

```powershell
.\run.ps1 start -RandomAttacks -IntervalSeconds 15
```

Остановка:

```powershell
.\run.ps1 stop
```

Просмотр логов отдельно:

```powershell
.\run.ps1 logs
```

## Флаги `start`

- `-Technique <T####>` — запуск одной конкретной атаки Atomic Red Team.
- `-RandomAttacks` — случайный выбор техники на каждом цикле.
- `-NoAttacks` — только фоновые логи, без атак.
- `-IntervalSeconds <N>` — пауза между циклами генерации и атаками; если не указать, выбирается случайно от 10 до 30 секунд.
- `-NoiseBatchSize <N>` — размер пакета фоновых логов за цикл; если не указать, выбирается случайно от 10 до 30.
- `-Detached` — запуск контейнера в фоне.

Только один режим атак может быть активен одновременно: `-Technique`, `-RandomAttacks` или `-NoAttacks`.

## Как это работает

Контейнер собран на полном Ubuntu-образе. Внутри образа установлены:

- Flog — генерация фонового Apache-шума
- Локальная копия `redcanaryco/atomic-red-team` с YAML-описаниями техник
- `pyyaml` — парсинг YAML-файлов техник

На каждом цикле генератор:

1. Случайно выбирает технику из списка доступных (или использует заданную).
2. Читает YAML-файл техники, извлекает команды из `atomic_tests`.
3. Извлекает **первое слово** команды (имя утилиты: `nmap`, `curl`, `powershell`, `python`, `ssh` и т.д.) — это называется **first-word dispatch**.
4. Ищет утилиту в словаре из 130+ известных команд и вызывает соответствующий handler, который генерирует 2-4 Apache syslog-строки.
5. Если утилита неизвестна — переходит к fallback по названию техники (ищет ключевые слова `scan`, `exploit`, `credential` и т.д.).
6. Если и это не сработало — техника считается непокрытой: записывается в `unknown_techniques.txt` и **пропускается** (не попадает в лог, таймлайн и маркеры).
7. Для покрытых техник фиксируются временные метки в `attack_timeline.log` и `attack_markers.csv`.

### Архитектура dispatch

```
first_word(command) → COMMAND_MAP[cmd_word]
                            ↓
                     нашёлся? → handler(command) → логи
                            ↓ нет
                     TECHNIQUE_FALLBACK[regex]
                            ↓
                     нашёлся? → handler(command) → логи
                            ↓ нет
                     → техника в unknown_techniques.txt, skip
```

**Покрытие:** ~85-90% техник MITRE ATT&CK получит релевантный лог. Непокрытые техники фиксируются для последующего анализа.

## Volume / папка логов на хосте

По умолчанию `run.ps1 start` создаёт в корне проекта папку `shared_external_logs` и монтирует её в контейнер как bind-mount. В этой папке вы увидите те же файлы, что и в внутреннем каталоге контейнера:

- `generated_logs.log` — все сгенерированные логи (шум + атаки)
- `attack_timeline.log` — таймлайн атак в формате `timestamp_start|timestamp_end|technique`
- `attack_markers.csv` — временные метки атак для сопоставления с анализом
- `unknown_techniques.txt` — техники без покрытия (накапливается между запусками)

## Примечания

- Симулятор обрабатывает все техники вне зависимости от платформы (Windows/Linux/macOS), так как генерирует только синтетические логи, не выполняя команды.
- При каждом запуске контейнера файлы `generated_logs.log`, `attack_timeline.log` и `attack_markers.csv` **очищаются**. Файл `unknown_techniques.txt` **не очищается** — накапливается между запусками.
- По умолчанию контейнер работает до остановки вручную.
